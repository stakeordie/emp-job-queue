# Service Integration Patterns: External vs Internal Services

This document explains the two architectural patterns used in the emp-job-queue system for integrating with services, comparing **External Service Integration** with **Internal Service Integration**.

## Pattern Overview

### External Service Integration
Workers connect to services hosted outside the machine - either cloud APIs (OpenAI, Replicate) or remote servers (remote ComfyUI, remote Playwright). The machine doesn't run the service locally.

### Internal Service Integration  
Workers connect to services running as processes on the same machine (local ComfyUI installation, local Ollama, local Automatic1111). The machine runs and manages the service processes.

## Architecture Comparison

### External Service Integration
```
┌─────────────────┐    Redis     ┌──────────────────┐    HTTPS     ┌─────────────────┐
│ Redis Job Queue │ ────────────→ │ Service          │ ───────────→ │ External        │
│                 │              │ Connector        │              │ Service         │
└─────────────────┘              │ (PM2 Process)    │              │ (Cloud/Remote)  │
                                 └──────────────────┘              └─────────────────┘
                                          │
                                          ▼
                                 ┌──────────────────┐
                                 │ Machine Status   │
                                 │ Aggregator       │
                                 └──────────────────┘

Examples: OpenAI API, Replicate API, Remote ComfyUI, Remote Playwright
```

### Internal Service Integration
```
┌─────────────────┐    Redis     ┌──────────────────┐   WebSocket   ┌─────────────────┐
│ Redis Job Queue │ ────────────→ │ Worker Service   │ ────────────→ │ Internal        │
│                 │              │ (PM2 Process)    │               │ Service         │
└─────────────────┘              └──────────────────┘               │ (PM2 Process)   │
                                          │                         └─────────────────┘
                                          ▼
                                 ┌──────────────────┐
                                 │ Machine Status   │
                                 │ Aggregator       │
                                 └──────────────────┘

Examples: Local ComfyUI, Local Ollama, Local Automatic1111
```

## Implementation Details

### External Service Integration Examples

#### OpenAI API Connector
```javascript
// Generated PM2 service by ecosystem-generator.js
{
  name: 'openai-connector',
  script: '/workspace/src/services/api-connectors/openai-connector.js',
  env: {
    OPENAI_API_KEY: process.env.OPENAI_API_KEY,
    CONNECTOR_TYPE: 'openai'
  }
}

// Connector implementation
export class OpenAIConnector extends BaseService {
  async processJob(jobData) {
    // Direct HTTP request to OpenAI API
    const response = await fetch('https://api.openai.com/v1/chat/completions', {
      headers: { 'Authorization': `Bearer ${process.env.OPENAI_API_KEY}` },
      body: JSON.stringify(jobData.prompt)
    });
    return await response.json();
  }
}
```

#### Remote ComfyUI Connector
```javascript
// Worker connects to ComfyUI running on different machine
export class ComfyUIWebSocketConnector extends BaseConnector {
  constructor(connectorId) {
    const host = process.env.WORKER_COMFYUI_HOST || 'comfyui.example.com';
    const port = process.env.WORKER_COMFYUI_PORT || '8188';
    const websocketUrl = `wss://${host}:${port}/ws`;
    
    super(connectorId, {
      base_url: websocketUrl,
      // No local service to manage
    });
  }
  
  async processJob(jobData) {
    // Connect to remote ComfyUI server
    return await this.sendToRemoteComfyUI(jobData.workflow);
  }
}
```

## Machine Type Usage

### API Machines (Direct API Pattern Only)
```javascript
// ecosystem-generator.js for API machines
function generateAPIServices() {
  const apiConnectors = process.env.MACHINE_API_CONNECTORS?.split(',') || 
    ['openai', 'replicate', 'runpod', 'weather'];
    
  apiConnectors.forEach(connector => {
    services.push({
      name: `${connector}-connector`,
      script: `/workspace/src/services/api-connectors/${connector}-connector.js`,
      env: {
        CONNECTOR_TYPE: connector,
        // Direct API credentials
        OPENAI_API_KEY: process.env.OPENAI_API_KEY,
        REPLICATE_API_TOKEN: process.env.REPLICATE_API_TOKEN
      }
    });
  });
}
```

### GPU Machines (Service Gateway Pattern Only)
```javascript
// ecosystem-generator.js for GPU machines
function generateComfyUIServices() {
  // 1. Application Services
  for (let gpu = 0; gpu < numGpus; gpu++) {
    services.push({
      name: `comfyui-gpu${gpu}`,
      script: '/workspace/src/services/comfyui/comfyui-service.js',
      env: { GPU_ID: gpu.toString() }
    });
  }
}

function generateRedisWorkerServices() {
  // 2. Worker Services with Connectors
  for (let gpu = 0; gpu < numGpus; gpu++) {
    services.push({
      name: `redis-worker-gpu${gpu}`,
      script: '/workspace/src/services/shared/redis-worker-service.js',
      env: {
        WORKER_CONNECTORS: 'comfyui,simulation',
        WORKER_COMFYUI_PORT: (8188 + gpu).toString()
      }
    });
  }
}
```

### Hybrid Machines (Both Patterns)
```javascript
// ecosystem-generator.js for hybrid machines
function generateEcosystem() {
```

## Job Processing Flow Comparison

### Direct API Pattern Flow
```
1. Job arrives in Redis queue
2. API Connector claims job directly
3. Connector makes HTTP request to external API
4. Connector processes response 
5. Connector publishes result to Redis
6. Connector reports status to Machine Status Aggregator

Timeline: ~1-10 seconds
Resources: Low CPU, network I/O
```

### Service Gateway Pattern Flow  
```
1. Job arrives in Redis queue
2. Redis Worker Service claims job
3. Worker loads appropriate connector (e.g., ComfyUI)
4. Connector establishes WebSocket connection to Application Service
5. Connector sends job data via WebSocket
6. Application Service processes job (heavy computation)
7. Application Service streams progress via WebSocket
8. Connector receives final result via WebSocket
9. Worker publishes result to Redis
10. Worker reports status to Machine Status Aggregator

Timeline: ~30 seconds - 10 minutes
Resources: High GPU, CPU, memory
```

## Pattern Selection Guidelines

### Use Direct API Pattern When:
- ✅ External service provides REST/HTTP API
- ✅ Processing is lightweight (< 30 seconds)
- ✅ No GPU/specialized hardware required
- ✅ Stateless operations
- ✅ Simple input/output format
- ✅ Want minimal resource overhead

**Examples**: OpenAI API, Replicate API, Weather APIs, Translation APIs, Text Processing

### Use Service Gateway Pattern When:
- ✅ Application requires persistent process
- ✅ Heavy computational workloads (> 30 seconds)
- ✅ GPU/specialized hardware required
- ✅ Complex state management needed
- ✅ Streaming/progress updates important
- ✅ Custom protocols (WebSocket, TCP, etc.)

**Examples**: ComfyUI, Automatic1111, Ollama, Local ML Models, Video Processing

## Status Reporting Integration

### Both Patterns Report to Same System
```javascript
// Machine Status Aggregator receives events from both patterns
async handleWorkerEvent(eventData) {
  const { worker_id, event_type, data } = eventData;
  
  switch (event_type) {
    case 'job_started':
      // Works for both Direct API and Service Gateway
      this.currentStatus.workers[worker_id].status = 'busy';
      this.currentStatus.workers[worker_id].current_job_id = data.job_id;
      break;
      
    case 'connector_status_changed':
      // Unified reporting regardless of pattern
      const serviceKey = `${worker_id}.${data.service_type}`;
      this.currentStatus.services[serviceKey].status = data.status;
      break;
  }
}
```

## Resource Usage Comparison

| Aspect | Direct API Pattern | Service Gateway Pattern |
|--------|-------------------|------------------------|
| **Memory Usage** | 50-200MB per connector | 1-8GB per service |
| **CPU Usage** | Low (HTTP processing) | High (ML/GPU processing) |
| **Startup Time** | 1-5 seconds | 30-120 seconds |
| **Concurrent Jobs** | 10-50 per connector | 1-5 per service |
| **Failure Recovery** | Fast (restart connector) | Slow (restart service) |
| **Scaling Strategy** | Horizontal (more connectors) | Vertical (more GPUs) |

## Error Handling Patterns

### Direct API Pattern Error Handling
```javascript
// Simple retry with exponential backoff
export class APIConnector extends BaseService {
  async processJob(jobData) {
    try {
      return await pRetry(() => this.callExternalAPI(jobData), {
        retries: 3,
        factor: 2,
        minTimeout: 1000
      });
    } catch (error) {
      // Fail fast - external API is down
      throw new ServiceError('External API unavailable', error);
    }
  }
}
```

### Service Gateway Pattern Error Handling
```javascript
// Complex recovery with service management
export class ServiceGatewayConnector extends BaseConnector {
  async processJob(jobData) {
    try {
      return await this.sendToService(jobData);
    } catch (error) {
      // Try to restart the application service
      await this.restartApplicationService();
      
      // Retry with fresh connection
      return await this.sendToService(jobData);
    }
  }
  
  async restartApplicationService() {
    // PM2 restart of heavyweight service
    await this.pm2.restart(`comfyui-gpu${this.gpu}`);
    await this.waitForServiceReady();
  }
}
```

## Migration Considerations

### Converting Direct API to Service Gateway
```javascript
// When an API becomes a heavyweight service:

// Before (Direct API)
class SimpleAIConnector extends BaseService {
  async processJob(job) {
    return await fetch(`https://api.service.com/process`, {
      method: 'POST',
      body: JSON.stringify(job.data)
    });
  }
}

// After (Service Gateway) 
class AIServiceConnector extends BaseConnector {
  async processJob(job) {
    // Now communicates with local AI service
    return await this.websocket.send({
      type: 'process',
      data: job.data
    });
  }
}
```

### Converting Service Gateway to Direct API
```javascript
// When a service becomes available as API:

// Before (Service Gateway)
class LocalModelConnector extends BaseConnector {
  async processJob(job) {
    // Complex WebSocket communication
    return await this.communicateWithLocalService(job);
  }
}

// After (Direct API)
class ModelAPIConnector extends BaseService {
  async processJob(job) {
    // Simple HTTP call to hosted API
    return await fetch(`https://model-api.com/infer`, {
      headers: { 'Authorization': `Bearer ${this.apiKey}` },
      body: JSON.stringify(job.data)
    });
  }
}
```

## Conclusion

The **Direct API Pattern** and **Service Gateway Pattern** serve different needs in the emp-job-queue system:

- **Direct API Pattern**: Optimal for lightweight, external API integrations with fast response times and low resource usage
- **Service Gateway Pattern**: Essential for heavyweight applications requiring persistent processes, GPU resources, and complex state management

Both patterns integrate seamlessly with the unified machine architecture, providing flexibility to handle diverse workload types while maintaining consistent job processing, status reporting, and error handling across the system.

Choose the pattern based on your integration requirements: external vs. local, lightweight vs. heavyweight, stateless vs. stateful.
